# Система моніторингу мережевих пристроїв — опис проєкту

## Що це і навіщо

Проєкт вирішує практичну задачу шкільного або офісного системного адміністратора: знати в реальному часі, які пристрої мережі живі, а які впали — і дізнаватися про це якомога швидше, без постійного ручного опитування. Система автоматично перевіряє доступність пристроїв через ICMP-ping, зберігає 24-годинну історію відгуків і відображає все у веб-інтерфейсі, який оновлюється миттєво без перезавантаження сторінки.

---

## Технологічний стек і аргументація вибору

### Python + FastAPI (backend)

Python обрано як мову з найбагатшою екосистемою для мережевих та системних задач. FastAPI — сучасний асинхронний веб-фреймворк, який надає три критично важливі речі одночасно: вбудовану підтримку WebSocket, автоматичну валідацію вхідних даних через Pydantic і автодокументацію API (Swagger UI за адресою `/docs`). Альтернативи — Flask або Django — не мають нативної асинхронності та WebSocket без додаткових бібліотек, що ускладнило б архітектуру.

### SQLite (база даних)

Для задачі хакатону та шкільної мережі масштаб не потребує PostgreSQL чи MySQL. SQLite — вбудована, безсерверна БД: файл `monitor.db` створюється автоматично при першому запуску, жодних зовнішніх залежностей. При цьому в коді використано всі «дорослі» практики: `PRAGMA foreign_keys = ON`, індекс на `(device_id, checked_at)` для швидкої вибірки історії, мьютекс `threading.Lock()` для безпечного запису з кількох потоків, `check_same_thread=False` для спільного з'єднання.

### Vanilla JavaScript + WebSocket (frontend)

Свідома відмова від React, Vue чи Angular. Причина: фронтенд рендерить список пристроїв, оновлює картки при зміні статусу і будує один графік. Це не виправдовує 200 KB+ фреймворку та систему збірки. Вся логіка — 453 рядки чистого JS із `esc()`-функцією для захисту від XSS при вставці даних у DOM. WebSocket замість HTTP-polling забезпечує миттєве оновлення (<50 мс затримки) і відсутність зайвих запитів: клієнт не питає «що нового?» кожну секунду — сервер сам надсилає зміни.

### Chart.js (графіки RTT)

Легка (60 KB), без залежностей, відмінно підходить для однотипного завдання — лінійний графік значень RTT по часу. Highcharts або D3.js були б надлишковими.

## Алгоритм опитування мережевих пристроїв

### Чому не Round-Robin з фіксованим інтервалом

Найпростіший підхід — опитувати кожен пристрій з однаковим інтервалом, наприклад раз на 60 секунд. Проблем дві: якщо пристрій щойно впав, ми дізнаємось про це в найгіршому випадку через 60 секунд; якщо пристрій стабільно працює тижнями — ми марно генеруємо трафік з тією ж частотою, що й для підозрілих вузлів.

### Обраний підхід: адаптивний інтервал

Система відстежує два лічильники для кожного пристрою: `consecutive_up` (скільки підряд успішних пінгів) і `consecutive_down` (скільки підряд невдалих). Після кожної перевірки функція `_next_interval()` визначає інтервал до наступного опитування за чотирма правилами:

```python
def _next_interval(cls, up: int, down: int) -> int:
    if down == 1:              return 15   # перший збій → перевірити швидше
    if down >= 2:              return 30   # підтверджений збій
    if up >= STABLE_THRESHOLD: return 120  # давно стабільний → рідше
    return 60                              # базовий стан
```

**Стан `unstable` (15 секунд)** — найважливіший момент алгоритму. Перший невдалий пінг не означає, що пристрій впав: мережевий пакет міг загубитися, сервер тимчасово завантажений. Замість негайно оголошувати пристрій `down` і будити адміністратора, система через 15 секунд робить ще одну перевірку. Тільки два підряд збоя підтверджують падіння. Це усуває хибні спрацьовування.

**Стан `down` (30 секунд)** — раз пристрій підтверджено не відповідає, немає сенсу пінгувати його щосекунди. 30 секунд — достатньо, щоб помітити відновлення без надмірного трафіку до недоступного хоста.

**Стан `stable` (120 секунд)** — якщо пристрій дав 5 підряд успішних пінгів (≈5 хвилин без проблем), він переходить у режим «рідкого опитування». Для шкільної мережі, де більшість серверів та комутаторів живі 99.9% часу, це значно знижує навантаження.

### Схема станів пристрою

```
         ping OK              ping OK (×5)
unknown ─────────► up ───────────────────► stable (120 с)
                   │  ◄──────────────────
                   │   будь-який збій
               ping FAIL
                   ▼
               unstable (15 с)
                   │
               ping FAIL (×2)
                   ▼
                 down (30 с)
                   │
               ping OK
                   ▼
                  up (60 с)
```

### Математика ефективності

При мережі з 10 стабільних і 2 нестабільних пристроїв за годину:

| Алгоритм | Запитів/год |
|----------|------------|
| Round-Robin 60 с | 720 |
| Адаптивний | ~540 |

Економія ~25% запитів. Водночас час виявлення першого збою скорочується з **до 60 с** до **15 с**.

### Реалізація: thread per device

Кожен пристрій має **власний** `threading.Timer` — незалежний потік, який самоперезапускається після кожної перевірки з новим інтервалом. Це принципово відрізняється від архітектури з одним «великим циклом», який по черзі перевіряє всіх: якщо один пристрій відповідає повільно (тайм-аут 3 секунди), він не затримує перевірку інших. Таймери — daemon-потоки, тому коректно завершуються разом із сервером.

Зв'язок між потоком поллера і async-циклом FastAPI реалізовано через `asyncio.run_coroutine_threadsafe()` — єдиний безпечний спосіб передати дані з синхронного потоку в асинхронний event loop без блокування.

## Сканування мережі

При додаванні пристроїв вручну потрібно знати IP. Для автоматизації реалізовано сканер підмережі: він визначає локальну IP-адресу сервера, будує CIDR `/24` (наприклад `192.168.1.0/24` — 254 хости), потім паралельно пінгує їх чанками по 20 через `asyncio.gather()`. Для кожного живого хоста робиться зворотний DNS-запит (`socket.gethostbyaddr`), а тип пристрою вгадується за двома факторами: ключові слова в hostname (`mikrotik` → router, `hikvision` → camera) і перевірка відкритих портів (9100/515 → принтер, 554 → IP-камера, 22 → сервер). Результат — список готових до імпорту пристроїв.

## Збереження даних і управління історією

Кожен результат опитування записується в таблицю `history` і одночасно очищуються записи старші 24 годин для цього пристрою. Це «ковзне вікно» — база не росте нескінченно, і не потрібен окремий cron-job для чищення.

Uptime розраховується прямо SQL-запитом:

```sql
SUM(CASE WHEN status = 'up' THEN 1 ELSE 0 END) / COUNT(*) * 100
```

Жодної логіки в Python — база сама рахує відсоток доступності за будь-який заданий часовий діапазон.

## Реальний час без перезавантаження

Коли поллер виявляє зміну статусу (`prev_status != new_status`), він викликає callback `_on_status_change`, який через `run_coroutine_threadsafe` передає повідомлення в WebSocket-менеджер. Менеджер обходить список усіх підключених клієнтів і надсилає JSON `{"type": "status_change", "device": {...}}`. Якщо з'єднання з якимось клієнтом вже закрито — воно видаляється зі списку автоматично. JavaScript на фронтенді оновлює лише картку конкретного пристрою, не перемальовуючи весь список.

## Реалізовані функції

### Обов'язкові
- Ф1 — Ping-перевірка доступності (ICMP)
- Ф2 — Панель моніторингу з переліком пристроїв та статусами
- Ф3 — Сповіщення в інтерфейсі при зміні статусу
- Ф4 — Додавання, редагування, видалення пристроїв
- Ф5 — Опис алгоритму опитування з обґрунтуванням

### Додаткові
- Д1 — Кольорова індикація: зелений / червоний / жовтий / сірий
- Д2 — Графік RTT за 24 години для кожного пристрою
- Д4 — Пошук та фільтрація за типом і статусом
- Д5 — Відсоток доступності за 24 години

### Понад специфікацію
- Автоматичне сканування підмережі з визначенням типу пристрою
- Адаптивний інтервал опитування (специфікація вимагала лише фіксований)
- Responsive дизайн (mobile-first, від 320px до 1920px)
