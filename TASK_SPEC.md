# Технічне завдання: Моніторинг мережевих пристроїв

## 1. Загальний опис

Розробити програму або веб-сервіс для автоматичного моніторингу стану мережевих пристроїв з відображенням їх статусу в реальному часі та сповіщенням про відмови.

---

## 2. Функціональні вимоги

### 2.1 Обов'язкові функції (мінімальний рівень)

| №  | Функція | Опис |
|----|---------|------|
| Ф1 | Опитування пристроїв | Автоматична перевірка доступності через ICMP-ping або SNMP |
| Ф2 | Панель моніторингу | Вебінтерфейс зі списком пристроїв та їх поточним статусом |
| Ф3 | Сповіщення | Повідомлення (інтерфейс, електронна пошта або Telegram) при зміні статусу пристрою |
| Ф4 | Управління пристроями | Додавання, редагування та видалення пристроїв зі списку |
| Ф5 | Опис алгоритму | Текстовий або графічний опис обраного алгоритму опитування з обґрунтуванням ефективності |

### 2.2 Додаткові функції (підвищений бал)

| №  | Функція | Опис |
|----|---------|------|
| Д1 | Кольорова індикація | Візуальне розрізнення статусів: зелений — активний, червоний — недоступний, жовтий — нестабільний |
| Д2 | Графіки доступності | Історія доступності пристрою у вигляді часового графіку |
| Д3 | Карта мережі | Топологічна схема мережі з відображенням зв'язків між пристроями |
| Д4 | Фільтрація та пошук | Пошук пристроїв за IP-адресою, назвою або типом |
| Д5 | Статистика | Відсоток часу доступності за обраний період (година/доба/тиждень) |

---

## 3. Типи пристроїв, що перевіряються

Система повинна підтримувати моніторинг щонайменше таких типів пристроїв:

- Сервери
- Маршрутизатори
- Комутатори
- Принтери
- Комп'ютери / робочі станції
- IP-камери

Кожен пристрій у системі має містити:

- IP-адресу або мережеве ім'я хоста
- Назву (довільний підпис)
- Тип пристрою
- Метод перевірки (ping / SNMP)
- Поточний статус
- Час останньої успішної відповіді

## 4. Алгоритм опитування

### 4.1 Вимога до документації алгоритму

Учасники **зобов'язані** описати та обґрунтувати обраний алгоритм опитування. Опис повинен включати:

1. Назву та принцип роботи алгоритму
2. Обґрунтування вибору (порівняння з альтернативами)
3. Оцінку часової складності та навантаження на мережу
4. Схему або псевдокод

### 4.2 Рекомендовані підходи 

- **Кругове опитування** — послідовне рівномірне опитування всіх пристроїв по черзі
- **Адаптивне опитування** — зміна частоти перевірки залежно від стабільності роботи пристрою
- **Опитування за пріоритетом** — критичні вузли перевіряються частіше за другорядні
- **Гібридний підхід (події + планове опитування)** — поєднання SNMP-пасток із регулярними перевірками

> Обрання гідридного підходу

## 5. Технічні вимоги

### 5.1 Технологічний стек

Учасники обирають стек самостійно. Рекомендовані варіанти:

| Рівень | Приклади |
|--------|---------|
| Серверна частина | Python (FastAPI, Flask), Node.js (Express), Go |
| Клієнтська частина | HTML/CSS/JS, React, Vue |
| База даних | SQLite, PostgreSQL, Redis |
| Опитування | `ping` (ICMP), `pysnmp`, `net-snmp` |

### 5.2 Вимоги до інтерфейсу

- Підхід **«спочатку мобільний»** з адаптивною версткою
- Коректне відображення на екранах від 320 пікселів до 1920 пікселів
- Елементи `section` та `div` — **без видимого контуру** (рамки чи обведення)
- Семантична HTML-розмітка
- Мінімальна кількість сторінок: 1 (інформаційна панель), можна більше
- Не додавати коментарі до коду
- Назви змінних, функцій і модулів зрозумілою мовою

### 5.3 Нефункціональні вимоги

- Час відгуку інтерфейсу — до 500 мілісекунд
- Автоматичне оновлення статусу без перезавантаження сторінки
- Зберігання мінімум 24 годин історії статусів


## 6. Обмеження та дозволи

- Дозволено використовувати бібліотеки та фреймворки з відкритим кодом
- Дозволено використовувати хмарні сервіси (Firebase, Render тощо) для розгортання
- Заборонено брати за основу готові рішення для моніторингу (Zabbix, Nagios, Grafana) — лише як джерело натхнення
- Весь код має бути написаний командою під час хакатону
