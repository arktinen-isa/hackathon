# Алгоритм опитування мережевих пристроїв

## Назва та принцип роботи

**Адаптивне інтервальне опитування (Adaptive Interval Polling)**

Кожен пристрій у системі має **власний незалежний таймер**. Після кожної перевірки система аналізує результат і динамічно розраховує інтервал до наступного опитування на основі поточної стабільності пристрою.

```
┌─────────────┐    перевірка     ┌────────────────────┐
│   Таймер    │ ──────────────►  │  ping / SNMP-запит │
└─────────────┘                  └────────────────────┘
       ▲                                  │
       │   розрахунок нового             ▼
       │   інтервалу             ┌────────────────────┐
       └──────────────────────── │  Аналіз результату │
                                 └────────────────────┘
```

### Правила зміни інтервалу

| Стан пристрою                     | Інтервал  | Обґрунтування                                       |
|-----------------------------------|-----------|-----------------------------------------------------|
| Початковий / невідомий            | 60 с      | Базове опитування                                   |
| Перший збій (`status = unstable`) | **15 с**  | Швидко підтвердити чи відхилити помилку             |
| Підтверджений збій (≥2 failures)  | **30 с**  | Пристрій вниз — часте опитування марне              |
| Стабільна робота (≥5 successes)   | **120 с** | Зменшити навантаження на мережу                     |

### Псевдокод

```
function calculate_next_interval(consecutive_up, consecutive_down):
    if consecutive_down == 1:   return 15   # перший збій → швидка перевірка
    if consecutive_down >= 2:   return 30   # підтверджений збій
    if consecutive_up   >= 5:   return 120  # дуже стабільний
    return 60                               # базовий стан

function poll(device):
    prev_status = device.status
    is_up, rtt  = ping(device.host)

    if is_up:
        device.consecutive_up   += 1
        device.consecutive_down  = 0
        device.status            = "up"
    else:
        device.consecutive_down += 1
        device.consecutive_up    = 0
        device.status = "down" if device.consecutive_down >= 2 else "unstable"

    device.poll_interval = calculate_next_interval(device.consecutive_up, device.consecutive_down)
    save_to_history(device, rtt)

    if prev_status != device.status:
        notify(device)

    schedule_next(device, delay=device.poll_interval)
```

## Порівняння з альтернативами

| Алгоритм                | Опис                                         | Проблема                                      |
|-------------------------|----------------------------------------------|-----------------------------------------------|
| **Round-Robin (фікс.)**  | Усі пристрої опитуються з однаковим інтервалом | Марнує ресурси на стабільні; повільно виявляє збої |
| **SNMP trap (пасивний)** | Пристрій сам надсилає сповіщення при збої    | Не всі пристрої підтримують; збої мовчки      |
| **Адаптивне (обрано)**  | Інтервал залежить від стабільності пристрою  | Мінімальний трафік + швидке виявлення збоїв   |
| **Priority-based**      | Критичні вузли опитуються частіше            | Потребує ручного призначення пріоритетів      |

## Оцінка ефективності

### Кількість запитів за 1 годину

Нехай мережа: 10 стабільних пристроїв + 2 нестабільних.

| Алгоритм              | Стабільні (10 × 1 год) | Нестабільні (2 × 1 год) | Разом |
|-----------------------|------------------------|--------------------------|-------|
| Round-Robin 60 с      | 10 × 60 = **600**      | 2 × 60 = **120**         | **720** |
| Адаптивний (120 с / 30 с) | 10 × 30 = **300**  | 2 × 120 = **240**        | **540** |

Економія: **~25 % запитів** у типовій мережі.  
Швидкість виявлення першого збою: **15 с** проти **до 60 с** у Round-Robin.

### Часова складність одного циклу

```
O(n) — де n = кількість пристроїв
```

Кожен пристрій опитується незалежно в окремому потоці (threading.Timer),  
тому збільшення кількості пристроїв не збільшує затримку для вже існуючих.

## Схема станів пристрою

```
          ping OK              ping OK (×5)
unknown ─────────► up ───────────────────► stable (120 с)
   │               │  ◄──────────────────
   │               │   будь-який збій
   │           ping FAIL
   │               ▼
   └──────────► unstable (15 с)
                   │
               ping FAIL (×2)
                   ▼
                 down (30 с)
                   │
               ping OK
                   ▼
                  up (60 с)
```
